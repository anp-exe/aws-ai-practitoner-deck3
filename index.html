03params
cost trap

Reducing Bedrock cost

tap to flip ✦
03params

Bedrock cost = tokens, only tokens

Cost is billed per million tokens, separately for input and output. Output is usually more expensive.

To reduce cost: fewer input tokens, fewer output tokens. That is the only lever.
Exam trap: Temperature, Top K, Top P, Stop sequences, batch size during training — NONE of these affect inference cost. They control behaviour, not billing.

Strategies that DO reduce cost: shorter prompts, shorter responses (max tokens limit), prompt caching, batch inference (~50% discount).

Bedrock Agents

Foundation models that take actions, not just generate text. They plan, call APIs, query databases, and complete multi-step tasks.

  • Bedrock manages prompt engineering, memory, monitoring
  • You define the tools and APIs they can invoke
  • Common pattern: insurance claims, travel bookings, inventory monitoring + reorder recommendations
Question keywords: "automate", "monitor and recommend", "execute multi-step task", "complete actions" → Agents.

Bedrock Knowledge Bases

Fully managed RAG. The FM answers using context retrieved from your private documents.

  • You connect a data source (S3, web crawler, Confluence, SharePoint)
  • Bedrock handles chunking, embeddings, vector storage, retrieval
  • Used when answers must come from YOUR data, not the model's training
Question keywords: "answer from internal docs", "contextual info from company data", "ground in private knowledge" → Knowledge Bases.

Knowledge Bases retrieves info; Agents take actions. If the question is about both (retrieve and then DO something), it's Agents (Agents can use KBs internally).

Bedrock Guardrails

Safety policies enforced on prompts AND responses. Five filter types:

  • Content filters: hate, violence, sexual, misconduct
  • Denied topics: custom topics you specify in natural language
  • Sensitive information filters: PII (block or mask), custom regex
  • Word filters: explicit blocklist
  • Contextual grounding checks: detect hallucinations vs source docs
Question keywords: "block harmful content", "redact PII", "responsible AI policy" → Guardrails.

Model Invocation Logging

Captures the full request and response data for every Bedrock model invocation, including input prompts, output text, and metadata.

  • Destinations: CloudWatch Logs or S3
  • Disabled by default, must be enabled per account/region
  • Used for audit, compliance, troubleshooting
Don't confuse with:
CloudTrail = logs API calls (who called what), NOT input/output data
EventBridge = triggers reactions to events, not detailed logs
AWS Config = monitors resource configurations, not data

Question keyword: "monitor input and output data", "audit prompts and responses" → Model Invocation Logging.

Provisioned Throughput

Reserves dedicated model capacity for guaranteed throughput and latency.

Critical exam fact: Provisioned Throughput is REQUIRED to deploy and use a fine-tuned or continued-pre-trained custom model in Bedrock. On-Demand mode does NOT support custom models.

Also useful when:

  • High-traffic production needs predictable performance
  • You hit throttling under default on-demand limits

Question keywords: "deploy fine-tuned model", "guaranteed throughput for production", "reserved capacity" → Provisioned Throughput.

Bedrock Playground

Console environment to test prompts and inference parameters on different models before building an app.

  • Text playground, chat playground, image playground
  • No code required
  • Used for prompt engineering experimentation
Common distractor: Playground CANNOT fine-tune models. It CANNOT deploy models to production. If a question is about fine-tuning, deployment, or running real workloads, Playground is wrong.

Transformer models

The architecture that powers modern LLMs (Claude, GPT, Llama).

  • Self-attention mechanism: weighs the importance of every token relative to every other token
  • Contextual embeddings: each word's representation depends on its context
  • Positional encodings handle word order
  • Encoder-decoder structure for sequence tasks
Exam phrase: "self-attention" + "contextual embeddings" → Transformer. Always.

Used for: language translation, summarisation, text generation, chat. Everything LLM is built on transformers.

Large payload OR >60s processing → Async
Huge dataset, no urgency → Batch

14services
AWS quirk!! exam trap

Document classification

tap to flip ✦
14services

AWS classifies it as SEMI-supervised

In general ML textbooks, document classification is supervised. AWS docs frame it as semi-supervised because real-world document collections are too large to fully label, so the typical approach uses a small labeled set + a large unlabeled set.

AWS examples by category:
Supervised: linear regression, neural networks, classification with labeled data
Unsupervised: clustering (k-means), association rule learning (Apriori)
Semi-supervised: document classification on large doc bases
Reinforcement: agent + reward signal

When in doubt on these "which learning type" questions, trust AWS's specific framing over your general ML intuition. The exam follows AWS docs.

"MOST effective" / "BEST" exam rule

When two answers are both technically correct, pick the one that most directly and precisely targets the stated problem.

Trigger phrases that activate this rule:

  • "MOST effective"
  • "BEST approach"
  • "PRIMARY"
  • "MOST suitable"
  • "MOST appropriate"
Worked example: Model is overfitting. "Add more training data" and "Hyperparameter tuning (regularization, dropout)" are BOTH valid fixes. But hyperparameter tuning is the direct intervention for overfitting; more data is indirect and depends on data quality. Answer: hyperparameter tuning.

Also: questions with this phrasing often have a "MOST cost-effective" variant. The rule applies the same way: pick the option that maps most precisely to the cost goal.
